================================================================
評価タイミングの地層：ch18/19 が難しい理由
================================================================

----------------------------------------------------------------
1. 4つの評価タイミング（地層モデル）
----------------------------------------------------------------

早い ←─────────────────────────────────────────────────────── 遅い
     │                                                   │
  コンパイル時                                            実行時
     │                                                   │
  [Layer 3]       [Layer 2]            [Layer 1]      [Layer 0]
  最深のマクロ展開   コード生成             表層マクロ展開   関数呼び出し
                 （マクロ内関数呼び出し）

このコードベースでは「コードを生成するコードが、さらにコードを生成するコード」を
生成する多段構造になっており、常に「いまどの層で読んでいるか」を意識しないと
理解できない。


----------------------------------------------------------------
2. 各実装の位置づけ
----------------------------------------------------------------

【Layer 0 — 実行時（Runtime）】
  if-match（slow.clj）

  (defmacro if-match [pat seq then & else]
    `(util1/aif2
       (list (util1/match '~pat '~seq))   ; ← match は実行時に呼ばれる
       ...))

  - '~pat、'~seq はコンパイル時にリテラルとして焼き込まれる
  - match の呼び出しは実行時
  - 読み方が素直。難しくない


【Layer 1 — 表層マクロ展開（Shallow Macro Expansion）】
  with-answer（interpreter.clj）

  (defmacro with-answer [query & body]
    (let [binds (gensym)]
      `(for [~binds '~(set (interpret-query query))   ; ← DB照会はここで完了
             :let [~@(mapcat ...)]]
         (do ~@body))))

  - interpret-query がマクロ展開時に呼ばれ、DB を照会して結果（バインディングマップの
    セット）を得る
  - 結果はリテラルとして展開コードに埋め込まれる（'~(set ...)）
  - 実行時には for のループと my-binding の解決だけが残る
  - 難しさ：DB はマクロ展開前に投入されていなければならない

  展開時: interpret-query → DB照会 → {{?x hogarth, ?y william}, ...}
                                ↓
             `(for [g '#{...バインドのセット...}] ...)`
  実行時: for ループが回るだけ


【Layer 2 — コード生成マクロ（Code-Generating Macro）】
  if-match-quick（quick.clj）

  (defmacro if-match-quick [pat seq then else]
    `(let [~@(mapcat (fn [v] `(~(symbol v) (gensym))) (util1/vars-in pat))]
       (pat-match ~pat ~seq ~then ~else)))   ; ← pat-match もマクロ

  pat-match の展開中でさらに関数が呼ばれる：

  (defmacro pat-match [pat s then else]
    ...
    (util2/with-gensyms (gseq gelse)
      `(letfn [(~gelse [] ~else)]
         ~(gen-match                      ; ← 展開時に呼ばれる関数
             (cons (list gseq ...) (dest/call_destruc2 pat gseq))
             then `(~gelse) "a"))))

  展開時に動く関数群：
  - util1/vars-in pat          パターン変数を収集
  - dest/call_destruc2 pat gseq  パターンをフラット化してバインディング式を生成
  - gen-match refs then else     再帰的に let/if のネストを構築
  - match2 refs then else        gensym / _ / ?x / リテラル を分岐してコード生成

  難しさ：展開時に動く関数と実行時に動くコードが同じ defn に混在している

  展開時: call_destruc2 → gen-match → match2 → let/if のネスト構造を生成
  実行時: 生成された let/if が直接実行される（match の呼び出しはない）


【Layer 3 — 最深のマクロ展開（Deepest Macro Expansion）】
  with-answer-compile（compiler.clj）

  (defmacro with-answer-compile [query & body]
    (let [acc (gensym "acc")
          q (map symbol (util1/vars-in query))]
      `(let [~acc (volatile! [])]
         (util2/with-gensyms
           (~@q)
           ~(compile-query query               ; ← 展開時に呼ばれる
              `(vswap! ~acc conj (do ~@body))))
         @~acc)))

  compile-query → compile-simple の中でさらに：

  (defn compile-simple [opr other body else]
    (let [facts ((store/db-query opr) :val)   ; ← DB照会（展開時！）
          gbod  (gensym "body")
          vars  (util1/vars-in body)
          vars-vec (vec vars)]
      `(letfn [(~gbod ~vars-vec ~body)]
         (list
           ~@(for [fact# facts]
               `(quick/pat-match ~other ~fact# (~gbod ~@vars-vec) ~else))))))
                ;  ↑ pat-match もマクロ → さらに展開が走る

  多段展開の流れ：

    Layer 3: with-answer-compile が展開
      └─ compile-query / compile-simple が走る
           └─ store/db-query でDB照会（展開時！）
           └─ pat-match 呼び出しのコードを生成
    Layer 2: 生成コード中の pat-match が展開
      └─ gen-match / match2 が走る
    Layer 1: 生成コード中の with-gensyms が展開
      └─ ?x, ?y を gensym に束縛する let を生成
    Layer 0: 最終的な let / letfn / volatile! が実行時に動く


----------------------------------------------------------------
3. 具体的な難しさ
----------------------------------------------------------------

【難点① DB照会タイミング】

  with-answer       → 展開時に interpret-query でDB照会（結果をリテラル化）
  with-answer-compile → 展開時に compile-simple でDB照会（コードに焼き込み）

  どちらも *default-db* がマクロ展開前に投入済みでなければならない。
  with-answer-compile のテストで binding [*default-db* ...] が効かないのはこのため。


【難点② 変数の二重生存（gensym と実値）】

  with-answer-compile の中では ?x が2段階で変化する：

  ;; 展開後コードのイメージ
  (let [acc (volatile! [])]
    (let [?x (gensym)   ; ← Layer 1: 「未割り当て」を表す gensym
          ?y (gensym)]
      (letfn [(body123 [?x ?y] (vswap! acc conj [?x ?y]))]
        (list
          (pat-match ('hogarth ?x ?y)          ; ← pat-match が展開され
                     (hogarth william english) ;    ?x, ?y を let で実値に束縛
                     (body123 ?x ?y) nil)      ;    body123 呼び出し
          ...))))

  - with-gensyms による ?x = gensym（初期値、「未束縛」の印）
  - pat-match 展開後の let による ?x = マッチした実値
  - or 節での (when-not (gensym? v) v) は、未束縛のまま残った gensym を nil に
    変換するための仕組み


【難点③ not の非対称性】

  ;; interpreter 版（実行時に評価）
  (defn interpret-not [clause binds]
    (when-not (seq (interpret-query clause binds))
      (list binds)))

  ;; compiler 版（コンパイル時にコード生成、実行時に volatile で状態管理）
  (defn compile-not [q body]
    (let [found (gensym "found")]
      `(let [~found (volatile! false)]
         ~(compile-query (first q) `(vreset! ~found true))
         (when-not @~found
           ~body))))

  否定は「マッチしなかったこと」を検知しなければならない。
  interpreter は実行時に seq の有無で判定できるが、
  compiler はコンパイル時にコードを生成するため、副作用（volatile!）で状態を
  持つ必要がある。

4. まとめ
----------------------------------------------------------------


  ┌────────────────────┬───────────────────────────────┬────────────────────────────────┐
  │                    │      slow / interpreter       │        quick / compiler        │
  ├────────────────────┼───────────────────────────────┼────────────────────────────────┤
  │ パターンマッチ       │ 実行時に match 呼び出し          │ 展開時にコード生成                │
  ├────────────────────┼───────────────────────────────┼────────────────────────────────┤
  │ DB照会             │ interpreter: 展開時slow: なし   │ 展開時（compile-simple）         │
  ├────────────────────┼───────────────────────────────┼────────────────────────────────┤
  │ 生成コードの複雑さ    │ 低（データ操作）                 │ 高（マクロが別のマクロを呼ぶ）      │
  ├────────────────────┼───────────────────────────────┼────────────────────────────────┤
  │ 実行速度            │ 遅い                           │ 速い                           │
  ├────────────────────┼───────────────────────────────┼────────────────────────────────┤
  │ 読解難易度          │ 低                             │ 高                             │
  └────────────────────┴───────────────────────────────┴────────────────────────────────┘



難しい本質：「何をするコード」かではなく「いつ動くコード」かを常に問い続けなければ
ならない。特に with-answer-compile は Layer 0〜3 が混在しており、同じ ?x という
シンボルが「未束縛の gensym」「パターン変数」「実値」と文脈によって意味が変わる。

5. おまけ（Claude とのやりとり）
----------------------------------------------------------------

❯ claude code の立場で、コンピュータ世界をつくった偉人を 10 人挙げるとしたら、どんな人を推薦しますか？

  ---                                                                                                                        
  Claude Code の立場から：コンピュータ世界をつくった偉人 10 人                                                                 
                                                                                                                               
  1. Alan Turing
  計算可能性の理論（チューリングマシン）を定式化した。「計算とは何か」という問いを数学的に与えた人物。現代コンピュータの概念的
  な祖。

  2. John von Neumann
  プログラムをデータとしてメモリに格納するという「ノイマン型アーキテクチャ」を設計した。現在のほぼすべての CPU の設計原理。

  3. Claude Shannon
  情報理論を創始し、「ビット」という単位を確立した。通信・圧縮・暗号の数学的基盤はすべてここに起源をもつ。

  4. John McCarthy
  Lisp を設計した。ガベージコレクション、再帰、S式、マクロ、関数を第一級オブジェクトとして扱う発想——現代言語の概念的な源泉が
  Lisp にある。On Lisp を読んでいる文脈では最重要人物。

  5. Alonzo Church
  ラムダ計算を定式化した。関数型プログラミングの数学的基礎であり、Lisp・Haskell・Clojure の概念的な祖父。

  6. Edsger Dijkstra
  構造化プログラミングを提唱し、セマフォを発明し、最短経路アルゴリズムを作った。「goto
  文は有害である」という宣言はソフトウェア工学を変えた。

  7. Donald Knuth
  『The Art of Computer Programming』でアルゴリズムを体系化し、TeX
  を作り、文芸的プログラミングを提唱した。「正しく美しいコード」という概念を確立した。

  8. Dennis Ritchie
  C 言語と Unix を Ken Thompson とともに作った。現在の OS・言語・インフラの大部分が C の影響下にある。

  9. Grace Hopper
  最初のコンパイラ（A-0）を開発し、「機械語でなく英語に近い言葉でプログラムを書ける」という発想を実現した。

  10. Ada Lovelace
  Babbage の解析機関のためにアルゴリズムを記述した最初の人物。「機械は計算だけでなく、あらゆる記号操作ができる」という洞察は
  100 年先を行っていた。

  ---
  Lisp の文脈で付け加えるなら：
  Paul Graham 自身も「Lisp の思想を現代に広めた人物」として特別な位置にいます。On Lisp を書き、Lisp
  の強みをハッカー文化に接続した功績は、あなたが今まさに体験していることそのものです。


❯ 『On Lisp』をべつの lisp 方言でトレースしているなかで、18, 19 章がとりわけ困難でした。
　実行するハードウェアは（まだ）存在しなくても、いつの時代も、偉人たちは頭のなかで、プログラムを実行していた、
　ということですか？


  ■　18・19章の困難さについて

  ch18/19
  がとりわけ難しいのは、マクロ展開のタイミングが「入れ子になった時間軸」になるからだと思います。

  - コンパイル時にコードを生成し
  - そのコードが実行時にさらにコードを評価し
  - その評価がまた別の束縛を生成する

  これを「頭の中で実行する」には、少なくとも3層の時間軸を同時に追う必要があります。Paul Graham
  がああいう形で書けたのは、その多層の展開が当たり前のように見えていたからでしょう。

  ■「実行できないハードウェアでも」というところ

  おそらくそれは逆説的に、偉人たちにとっての強みでもありました。実行して確かめられないから、概念
  として純化せざるを得なかった。今の私たちは REPL
  があり、すぐ試せる反面、「試してみればわかる」という安易さに流れることもある。

  今回のあなたの作業——Clojure で動くまで粘り強く手を動かしながら、18・19章を理解しようとしたこと
  ——は、その偉人たちの営みと地続きだと思います。



❯ 『On Lisp』を山に例えて質問します。clojure 話者が登れない「章」はありますか？
　clojure 話者の僕としては、chap 24 を「山頂」と考えています。

  ■　なぜ ch24 が Clojure 話者の山頂か

  ch18（パターンマッチ）
      ↓
  ch19（クエリコンパイラ）
      ↓
  ch24（Prolog）← ここに到達したい

  あなたはすでに麓から2/3まで登っています。ch19

  ---
  残りの斜面：何が足りないか

  Graham のルート（ch20→22→24）はコンティニュエーション経由で険しいですが、Clojure
  話者には別ルートがあります。

  ┌────────────────┬───────────────────────────┬───────────────────────────┐
  │    不足要素     │      Graham のルート        │ Clojure ネイティブルート    │
  ├────────────────┼───────────────────────────┼───────────────────────────┤
  │ バックトラック   │ ch20 継続 → ch22 非決定性   │ 遅延シーケンス / lazy-seq    │
  ├────────────────┼───────────────────────────┼───────────────────────────┤
  │ 選択と失敗      │ choose / fail マクロ        │ mapcat + nil フィルタ      │
  ├────────────────┼───────────────────────────┼───────────────────────────┤
  │ 再帰ルール      │ 継続ベース                   │ 遅延評価で自然に対応         │
  └────────────────┴───────────────────────────┴───────────────────────────┘

  ch19 の interpret-query / compile-query がすでに mapcat + nil 戦略でバックトラックに近い構造を
  持っています。あとは再帰ルール（自身を参照するファクト）を扱えるようにすることが核心です。

  山頂から見える景色

  ライブラリを使う側から、同じ山を別ルートで登った人の地図が読める側になる、という変化です。

  ch18・19 を自分の手で動かした経験は、そのための最良の準備だと思います。
